# 開発の進め方（ブランチ & プルリク運用）

このリポジトリ（カロリーダッシュボード）を、実務と同じ **ブランチ → PR → レビュー → マージ** の流れで育てるためのメモ。

## 全体像

```
master(本番) ──●──────────────●(merge)────► GitHub Pagesが自動公開
                \            /
   feature ブランチ ●──●──●(PR)   ← ここで作って、テストして、レビューして本番へ
```

- **master** = 本番。GitHub Pages が公開している唯一の枝（https://kiyohirokouno1986.github.io/）。
- **feature ブランチ** = master のコピーで作る作業用の枝。ここをいじっても本番は変わらない。
- **1機能 = 1ブランチ = 1PR**。小さく出して、レビューして、混ぜる。

## 用語チートシート

| 用語 | 意味 |
|---|---|
| commit | 変更の保存単位（セーブポイント）。メッセージで「何をしたか」を残す |
| branch | 並行して作業するための枝。`master` から分岐させる |
| push | 手元の commit を GitHub に送る |
| PR (Pull Request) | 「この差分を master に入れたい」という提案書。差分(diff)とレビューの場 |
| diff | 変更前後の差分。`+` が追加、`-` が削除 |
| review | 差分を見て可否を判断すること |
| merge | 承認して master に取り込むこと。ここで本番が変わる |
| deploy | 本番反映。master が変わると GitHub Pages が自動で再公開 |
| draft PR | 下書き状態のPR。まだマージ対象でない「作業中」の印 |

## 1サイクルの手順

1. **ブランチを切る**：`master` の最新から feature ブランチを作成（例 `feature/overflow-diagnosis`）。
2. **実装 → commit → push**：変更して保存し、GitHub に送る。
3. **ドラフトPRを作成**：差分が見える状態にする。
4. **テスト**：ブランチの状態を動かして確認（スクショ・検証結果をPRに添付）。
5. **レビュー**：差分とスクショを確認。疑問はPR上でやり取り。
6. **マージ**：OKなら `Merge` ボタンで master へ。→ Pages が自動デプロイ。
7. **本番確認**：公開URLを強制リロードして最終チェック。
8. **後片付け**：使い終わった feature ブランチを削除。

## ブランチ名の付け方（ゆるい規約）

- `feature/xxx` … 新機能（例 `feature/overflow-diagnosis`）
- `fix/xxx` … バグ修正（例 `fix/pfc-parser`）
- `docs/xxx` … ドキュメント（例 `docs/dev-workflow`）
- `chore/xxx` … 雑務・設定（例 `chore/cleanup`）

## レビューのやり方（PR画面で見るところ）

- **Files changed** タブ：差分そのもの。`+`（緑）が追加、`-`（赤）が削除。
- **Conversation** タブ：説明・スクショ・やり取り。
- 良ければ右上の **Merge pull request**（このリポジトリでは squash merge 推奨＝コミットを1つにまとめて綺麗に保つ）。
- マージ後、ブランチ削除の案内が出たら **Delete branch** でOK。

## プレビュー環境について（今後の発展）

GitHub Pages は **master しか公開しない**ので、feature ブランチは自動ではURLにならない。
当面はサンドボックスでの**動作スクショ＋検証結果をPRに添付**してプレビュー代わりにする。

将来、**PRごとに自動で一時URL**が欲しくなったら、Cloudflare Pages / Netlify を繋ぐと実現できる
（実務でいう「プレビュー環境」）。必要になったタイミングで導入する。
